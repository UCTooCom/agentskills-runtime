# 技术设计原则

## 1. 架构设计原则

### 1.1 分层架构
- **Controllers 层**: 处理 HTTP 请求，参数验证，响应封装
- **Services 层**: 业务逻辑处理，数据操作
- **Models 层**: 数据模型定义，ORM 映射
- **Routes 层**: 路由定义，中间件配置

### 1.2 模块化设计
- 按数据库和表名组织模块
- 每个模块独立、可测试
- 遵循单一职责原则

### 1.3 依赖注入
- 通过构造函数注入依赖
- 便于单元测试和模块替换

## 2. 代码规范

### 2.1 命名规范
- **类名**: PascalCase (如 `AgentSkillsController`)
- **方法名**: camelCase (如 `getSkillById`)
- **常量**: UPPER_SNAKE_CASE (如 `MAX_RETRY_COUNT`)
- **文件名**: PascalCase (如 `AgentSkillsController.cj`)

### 2.2 注释规范
- 类和公共方法必须有文档注释
- 复杂逻辑必须有行内注释
- 使用标准注释格式

### 2.3 错误处理
- 使用统一的错误响应格式
- 记录详细的错误日志
- 提供有意义的错误消息

## 3. API 设计规范

### 3.1 RESTful 规范
- 使用标准 HTTP 方法 (GET, POST, PUT, DELETE)
- 使用复数名词表示资源
- 使用嵌套表示资源关系

### 3.2 URL 规范
- 基础路径: `/api/v1/{database}/{table}`
- 操作路径: `/api/v1/{database}/{table}/{action}`
- 资源路径: `/api/v1/{database}/{table}/{id}`

### 3.3 响应格式
```json
{
  "data": {},
  "currentPage": 1,
  "totalCount": 100,
  "totalPage": 10
}
```

### 3.4 错误响应格式
```json
{
  "errno": "40001",
  "errmsg": "参数错误"
}
```

## 4. 数据库设计规范

### 4.1 表命名
- 使用小写字母和下划线
- 使用复数形式 (如 `agent_skills`)

### 4.2 字段命名
- 使用小写字母和下划线
- 主键统一使用 `id`
- 时间字段使用 `created_at`, `updated_at`, `deleted_at`

### 4.3 软删除
- 使用 `deleted_at` 字段实现软删除
- 查询时默认过滤已删除记录

## 5. 安全规范

### 5.1 认证
- 使用 JWT 进行身份认证
- Token 有效期合理设置

### 5.2 权限
- 实现基于角色的访问控制
- 敏感操作需要权限验证

### 5.3 数据验证
- 输入数据必须验证
- 防止 SQL 注入
- 防止 XSS 攻击

## 6. 性能优化

### 6.1 数据库优化
- 合理使用索引
- 避免 N+1 查询
- 使用分页查询

### 6.2 缓存策略
- 合理使用缓存
- 设置合适的缓存过期时间
- 缓存失效策略

## 7. 测试规范

### 7.1 单元测试
- 测试覆盖率 > 80%
- 测试边界条件
- 测试异常情况

### 7.2 集成测试
- 测试 API 接口
- 测试数据库操作
- 测试中间件

## 8. 文档规范

### 8.1 代码文档
- API 文档使用 OpenAPI 规范
- 数据库文档使用 ER 图

### 8.2 变更文档
- 记录所有变更
- 维护版本历史
