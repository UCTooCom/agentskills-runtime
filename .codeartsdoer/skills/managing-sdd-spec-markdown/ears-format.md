# EARS 格式规范

## 概述
EARS (Easy Approach to Requirements Syntax) 是一种简化的需求语法格式，用于编写清晰、可测试的需求。

## EARS 模式

### 1. 普遍性需求 (Ubiquitous)
**格式**: `系统应[功能描述]`

**示例**:
- 系统应提供用户登录功能
- 系统应支持多种数据库连接

### 2. 事件驱动需求 (Event-driven)
**格式**: `当[触发条件]时，系统应[功能描述]`

**示例**:
- 当用户点击登录按钮时，系统应验证用户凭证
- 当数据库连接失败时，系统应记录错误日志

### 3. 状态驱动需求 (State-driven)
**格式**: `当[系统状态]时，系统应[功能描述]`

**示例**:
- 当系统处于维护模式时，系统应拒绝所有新请求
- 当用户未登录时，系统应重定向到登录页面

### 4. 可选功能需求 (Optional)
**格式**: `当[可选功能]启用时，系统应[功能描述]`

**示例**:
- 当缓存功能启用时，系统应缓存查询结果
- 当日志记录启用时，系统应记录所有操作

### 5. 复杂需求 (Complex)
**格式**: `当[条件1]且[条件2]时，系统应[功能描述]`

**示例**:
- 当用户已登录且具有管理员权限时，系统应允许访问管理面板

## 验收标准编写规范

### 格式
每个需求应包含：
1. **需求ID**: REQ-XXX-NNN (模块-序号)
2. **需求描述**: 使用 EARS 格式
3. **验收标准**: 可测试的具体条件
4. **优先级**: P0(必须) / P1(重要) / P2(一般)

### 示例

```markdown
#### REQ-API-001: 用户登录认证

**需求描述**: 当用户提交登录表单时，系统应验证用户名和密码的正确性。

**验收标准**:
- [ ] 输入正确的用户名和密码，登录成功
- [ ] 输入错误的密码，显示"密码错误"提示
- [ ] 输入不存在的用户名，显示"用户不存在"提示
- [ ] 连续失败3次，锁定账户15分钟

**优先级**: P0
```

## 需求分类

### 功能性需求
描述系统应该做什么，使用 EARS 格式编写。

### 非功能性需求
描述系统的质量属性：
- 性能需求
- 安全需求
- 可用性需求
- 可维护性需求

### 约束性需求
描述系统的限制条件：
- 技术约束
- 业务约束
- 法规约束
