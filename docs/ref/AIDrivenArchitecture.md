# 架构师观点
1. 不硬编码业务功能到系统中，所有功能均可通过可视化配置实现动态调整
2. 基于agentskills一等公民的设计原则，以及agentskills自身特性，其实可以省略开发框架中很多确定性运行的代码，系统中的很多功能都可以由大模型推理和自主决策实现。但是，agentskills-runtime需要实现的是双驱动开发框架，即一方面AI可驱动开发框架实现自动化和效率提升，另一方面作为一个完备的开发框架在没有AI的情况下也可以人工操作使用开发框架的全部功能。以大模型为核心的harness在缺少AI的情况下就没有太多用处了，而AI驱动的开发框架没有AI依然是一个完备的服务提供方。另外，确定性运行的代码可以在很多情况下节省大模型的token。

# AI的观点分析
这两个观点触及了 AI 驱动开发框架的核心设计哲学，我来逐条客观分析：

观点 1 分析："不硬编码业务功能，所有功能可视化配置"
合理之处：

符合低代码/零代码平台趋势，降低定制成本
与 agentskills-runtime 已有的 db_info 驱动代码生成（crudgen/crudweb）一脉相承
数据库表结构本身已是"配置"，新增实体无需写代码即可生成全栈 CRUD
需警惕的边界：

"所有"过于绝对——安全机制（权限校验、JWT 认证）、同步引擎（SyncManager 循环防护）、调度器（SchedulerEngine 超时/重试）等基础设施逻辑不适合配置化，配置化会引入不可控的运行时风险
配置化系统自身也需要"硬编码"的引擎来解释和执行配置，这个引擎无法配置化自身
过度配置化会导致"配置地狱"——配置的复杂度超过代码的复杂度，调试和排错反而更困难
建议修正：区分业务功能层（可配置化）和基础设施层（需硬编码保障确定性）。agentskills-runtime 的分层架构天然支持这个区分：src/app/controllers/services/dao 是业务层可配置化，src/core/agent/skill/tool/model 和 src/app/services/sync/crontab 是基础设施层需确定性。

观点 2 分析："双驱动开发框架 vs Harness"
这是两个观点中更有深度和原创性的，值得重点分析。

"省略确定性代码，由大模型推理替代"
合理之处：

技能编排、任务分解、Agent 选择等确实可以由大模型动态决策，而非硬编码 if-else
AgentGroup 的 DSL 运算符（<=、|>、|）已体现了"声明意图而非执行步骤"的理念
PlanReactExecutor 的"先规划再执行"本身就是让大模型替代硬编码的执行计划
需警惕的边界：

token 成本：观点自身也承认"确定性代码可以节省 token"。每次让大模型重新推理一个可确定的结果，是 token 浪费。例如 CRUD 路由注册、权限校验、SQL 查询构建，这些确定性逻辑用代码实现比让大模型推理高效 100 倍以上
延迟与可靠性：大模型推理有延迟（秒级）和不确定性（相同输入可能不同输出），确定性代码是毫秒级且 100% 可预测。关键路径（如 SyncInterceptor 循环防护、权限校验）不能用大模型推理替代
可测试性：确定性代码可单元测试，大模型推理结果不可重复，难以回归测试
"双驱动：有 AI 是 AI 驱动，无 AI 仍是完备框架"
这是整个观点中最核心的洞察，也是 agentskills-runtime 区别于 Harness 的根本定位差异：


code
 
Harness (如 OpenClaw/Claude Code):
  AI 存在 → 完整功能
  AI 缺失 → 几乎无用（只剩 CLI 工具壳）

AI 驱动开发框架 (agentskills-runtime):
  AI 存在 → AI 驱动自动化 + 全部框架能力
  AI 缺失 → 仍是完备的服务提供方（CRUD API + 权限 + 同步 + 定时任务 + ...）
这个洞察的正确性可从三个维度验证：

商业维度：企业不会将核心业务系统建立在"没有 AI 就瘫痪"的架构上。Harness 是开发工具，AI 驱动开发框架是生产系统，两者定位不同
可靠性维度：AI 不可用时（模型服务故障、API 限流、网络中断），系统必须能继续提供基础服务。这是生产级系统的基本要求
渐进式采纳维度：用户可以从"无 AI 的传统框架"起步，逐步引入 AI 能力，而非一次性全押 AI。这大幅降低了采纳门槛
"Harness 未来的演进方向就是 AI 驱动开发框架"
部分同意，但需补充：

Harness 的演进确实在向"框架化"发展（Claude Code 已有 54 个工具、子 Agent 编排、记忆系统，这些是框架组件而非纯 AI 推理）
但 Harness 和 AI 驱动开发框架的设计原点不同：Harness 从 AI 出发向下沉淀确定性能力，AI 驱动开发框架从确定性框架出发向上叠加 AI 能力。两者在中间某处会合，但起点决定了基因：
Harness 基因：AI 优先，确定性是 AI 的脚手架
AI 驱动开发框架基因：确定性优先，AI 是确定性的增强器
agentskills-runtime 的架构（PostgreSQL 为中心 + 文件系统双向同步 + CRUD 全栈生成 + 权限体系）是典型的"确定性框架基因"，AI（CangjieMagic Agent/Skill）是叠加层。这个基因排序是正确的。

核心建议：在 AIDrivenArchitecture.md 中补充一条原则——"确定性优先，AI 增强"原则：凡是可确定性实现的逻辑（CRUD、权限、同步、调度），用代码实现；凡是需要推理、判断、创造的逻辑（任务分解、技能选择、代码生成），由 AI 驱动。两者不是替代关系，而是分层协作关系。这既是节省 token 的工程实践，也是"双驱动"框架的架构根基。


## AI驱动的全栈开发框架最佳实践    
1. **统一技术栈原则**：优先采用可支持一致技术栈实现全平台应用开发的编程语言，便于应用统一的编程规范和降低AI学习成本。  
2. **Monorepo架构优化**：采用monorepo组织工程的代码管理方案，便于AI IDE及大模型全面获取工程信息，实现更精准的代码生成。 
3. **CLI友好设计**：开发框架功能均可通过CLI命令行工具实现，便于与AI IDE及大模型的集成和互操作，支持自动化工作流。  
4. **AI Native原则**：原生智能化、天生全场景、高性能、强安全，符合AI Native的发展趋势，内置AI增强功能。  
5. **模型同构规范**：遵循UMI全栈模型同构规范，为确定性代码生成和AI辅助代码生成提供良好基础设施。  
6. **AI生态集成**：支持MCP、AgentSkills、WebMCP等AI生态主流协议和开发范式，确保与主流AI工具链的兼容性。

对生成结果提供“决策依据”或“来源链接”，便于人工复核与故障排查。
13. 依赖与模型版本治理
明确模型与依赖的版本矩阵；设灰度/金丝雀发布策略，允许按项目/环境快速切换模型。
给AI工具链（代码生成器、Lint、格式化器）也做版本锁定与变更日志。
14. 领域模型优先与契约驱动
先固化Domain Model与API契约，再驱动AI生成实现与前后端类型；用OpenAPI/JSON Schema/Protobuf做单一事实源。
通过代码生成器在多端同步类型，减少手写与漂移。
15. Prompt 即基础设施
将高价值Prompt抽象为“可测试、可发布、可版本化”的资源（如以文件/配置管理），支持回滚与A/B。
为关键Prompt建立性能基准与质量门槛（正确率、覆盖率、延迟）。
16. 人工评审与自动质量闸
在CI中设质量闸：静态检查、测试覆盖、Bundle大小、性能阈值、可访问性分数。
关键变更（安全、支付、数据导出）必须双人评审；AI建议仅作参考，不可越权合并。
17. 开发者体验与脚手架
提供一键初始化脚手架：内置目录规范、环境变量模板、示例Prompt、测试与CI配置。
常见业务基元（列表-详情、搜索-分页、表单-校验、权限-菜单）沉淀为可复用蓝图供AI调用。
18. 渐进式引入与风险控制
从低风险区域开始引入AI（文档、测试、样式、重复性脚本），逐步扩展到核心业务。
为AI生成变更设置隔离分支与特性开关；关键路径保留人工Fallback。
19. 性能与成本优化
在前端采用增量编译、按需加载、边缘缓存；在后端引入连接池、批处理、幂等重试。
对AI调用做缓存与去重；统一计费与用量监控，设超额预警与熔断策略。
20. 多环境一致性与基础设施即代码
Dev/Staging/Prod 配置分离且可追溯；用IaC（如Terraform）管理云资源与权限。
数据库迁移（如Prisma/Knex/Flyway）与Seed数据自动化，确保环境可复现。
