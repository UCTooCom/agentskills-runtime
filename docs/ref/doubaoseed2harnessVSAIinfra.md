## 概述

在当前AI智能体（AI Agent）领域，存在两种截然不同的技术思路和架构理念：

1. **Harness Engineering思路**：以大模型厂商为代表，强调模型能力的最大化释放，将Harness视为包裹大模型的基础设施层。代表项目包括Anthropic的Claude Code、OpenAI的Agents SDK、LangChain的LangGraph等。

2. **AI驱动的全栈开发框架思路**：以软件厂商为代表，强调整体软件系统的综合考虑，将AI大模型视为系统中的一个驱动子系统，而非核心。代表项目包括UCToo、AgentSkills Runtime等。

本文档将对这两种思路进行深度对比分析，从多个维度探讨其优势、劣势、适用场景，以及未来发展趋势。

## 两种思路的核心概念

### 1. Harness Engineering思路

#### 核心理念
> "裸LLM是没有内存、没有硬盘、没有I/O的CPU。Harness就是操作系统。"
> —— Beren Millidge

#### 关键定义
- **Agent**：涌现出来的行为，是用户交互的那个有目标、会用工具、能自我纠正的实体
- **Harness**：产生Agent行为的机器，是包裹LLM的完整软件基础设施
- **三个工程层级**：
  - 提示词工程：负责设计模型接收的指令
  - 上下文工程：管理模型看到什么、什么时候看到
  - Harness工程：包含前两者，再加完整的应用基础设施

#### 生产级Harness的12个组件
1. **编排循环**：思考-行动-观察（TAO）循环，也叫ReAct循环
2. **工具**：Agent的手，以schema形式定义
3. **记忆**：短期记忆（单次会话）、长期记忆（跨会话）
4. **上下文管理**：防止上下文腐烂，采用压缩、观察屏蔽、即时检索等策略
5. **提示词构建**：分层构建：系统提示、工具定义、记忆文件、对话历史、当前用户消息
6. **输出解析**：原生工具调用，schema约束响应
7. **状态管理**：类型化状态、检查点、时间旅行调试
8. **错误处理**：瞬时错误重试、LLM可恢复错误、用户可修复错误、意外错误
9. **护栏与安全**：输入护栏、输出护栏、工具护栏、断路器机制
10. **验证循环**：基于规则的反馈、视觉反馈、LLM作为评判者
11. **子agent编排**：Fork、Teammate、Worktree模式
12. **（原文未明确列出第12个，按上下文推测为性能监控）**

#### 代表项目特性
- **Claude Code**：
  - 「最小脚手架、最大操作Harness」设计哲学
  - AI决策逻辑仅占约1.6%
  - 七层安全机制：工具预过滤、拒绝优先规则、权限模式、ML分类器、沙箱隔离、不继承旧权限、Hooks拦截
  - 五层上下文压缩：预算裁剪、历史修剪、微压缩、上下文折叠、自动摘要
  
- **OpenAI Agents SDK**：
  - Runner类实现Harness
  - 支持异步、同步、流式三种模式
  - 代码优先：工作流逻辑用原生Python表达
  
- **LangChain/LangGraph**：
  - 将Harness建模为显式状态图
  - 节点（llm_call、tool_node）通过条件边连接
  - 从AgentExecutor演进而来

### 2. AI驱动的全栈开发框架思路

#### 核心理念
> "统一技术栈、Monorepo架构、CLI友好设计、AI Native原则、模型同构规范、AI生态集成"
> —— UCToo最佳实践

#### 关键定义
- **全栈模型同构(UMI)**：在分布式系统间一致性同步状态的设计规范，实现前端应用（客户端缓存或客户端数据库）与服务端应用（云端数据库）之间高效便捷的数据同步
- **Monorepo**：采用单一代码仓库管理整个项目，便于AI IDE及大模型全面获取工程信息
- **AI Native**：原生智能化、天生全场景、高性能、强安全，符合AI原生的发展趋势
- **渐进式AI增强**：支持平滑过渡到AI辅助开发，在原有项目中渐进式实现"+AI"能力

#### UCToo技术体系的15条最佳实践
1. **统一技术栈原则**：优先采用可支持一致技术栈实现全平台应用开发的编程语言
2. **Monorepo架构优化**：采用monorepo组织工程的代码管理方案
3. **CLI友好设计**：开发框架功能均可通过CLI命令行工具实现
4. **AI Native原则**：原生智能化、天生全场景、高性能、强安全
5. **模型同构规范**：遵循UMI全栈模型同构规范
6. **AI生态集成**：支持MCP、AgentSkills、WebMCP等AI生态主流协议
7. **人机协同开发流程**：以"人设目标、AI产草稿、人做评审与合并"为主线
8. **统一编码规范**：Commitlint、分支策略、目录结构、提示词模板
9. **可测试的AI产出**：单测/集成测试/契约测试覆盖AI生成模块
10. **强化安全与合规**：SAST/DAST/依赖漏洞扫描、Secrets扫描
11. **数据与隐私治理**：敏感数据最小化出境、数据可追溯链路
12. **观测与可解释性**：接入日志、指标、分布式追踪
13. **依赖与模型版本治理**：明确版本矩阵、灰度发布策略
14. **领域模型优先与契约驱动**：先固化Domain Model与API契约
15. **Prompt即基础设施**：将高价值Prompt抽象为可测试、可发布、可版本化的资源

#### AgentSkills Runtime架构特性
- **三层架构**：Controller层（表示层）、Service层（应用层）、Repository层（基础设施层）
- **核心模块**：magic.app（主应用）、magic.core（核心领域）、magic.skill（技能管理）、magic.model（AI模型集成）
- **关键能力**：
  - WASM安全沙箱
  - 高级RAG搜索（混合密集+稀疏搜索）
  - 多语言SDK（JavaScript/TypeScript、Python、Java、Go、Rust、C#）
  - MCP（Model Context Protocol）集成
  - 完整的内置工具集（文件系统、网络、技能、代码生成、CLI）
  - RBAC权限体系

## 详细对比分析

### 维度1：系统架构视角

#### Harness Engineering思路
```
┌─────────────────────────────────────┐
│         用户交互层                   │
├─────────────────────────────────────┤
│         Harness层                   │
│  ┌─────────────────────────────┐  │
│  │ 编排循环  工具  记忆  上下文│  │
│  │ 状态管理  错误处理  安全护栏│  │
│  └─────────────────────────────┘  │
├─────────────────────────────────────┤
│         LLM大模型层                 │
└─────────────────────────────────────┘
```

**特点**：
- LLM大模型处于系统核心位置
- Harness层包裹LLM，提供各种基础设施
- 系统设计围绕如何最大化释放LLM能力
- 强调"笨循环"，智能都在模型里

#### AI驱动全栈开发框架思路
```
┌─────────────────────────────────────────┐
│  前端层  │  移动端  │ 小程序  │ Web  │
├─────────────────────────────────────────┤
│         API网关层                      │
├─────────────────────────────────────────┤
│  业务服务层  │  技能运行时  │ 工具集│
├─────────────────────────────────────────┤
│  领域模型层  │  AI子系统  │  RAG    │
├─────────────────────────────────────────┤
│  数据层  │  缓存层  │  消息队列  │     │
└─────────────────────────────────────────┘
```

**特点**：
- 遵循传统软件架构的分层设计
- AI只是其中一个驱动子系统
- 强调整体系统的可维护性、可扩展性
- 注重领域模型、API契约、数据一致性

### 维度2：技术栈选择

#### Harness Engineering思路
- **编程语言**：Python为主（OpenAI、LangChain），TypeScript为辅（Claude Code）
- **核心框架**：LangChain、LangGraph、OpenAI Agents SDK、Claude Agent SDK
- **依赖**：重度依赖LLM API调用
- **部署**：通常作为服务或SDK集成到现有系统

#### AI驱动全栈开发框架思路
- **编程语言**：仓颉（国产自主可控）、TypeScript、ArkTS、Go等
- **核心框架**：Fountain ORM、自定义HTTP框架、AgentSkills Runtime
- **依赖**：平衡LLM调用与传统软件组件
- **部署**：完整的应用服务器、数据库、缓存等全栈部署

---

### 维度3：开发模式

#### Harness Engineering思路
1. **提示词驱动**：大量精力花在提示词工程上
2. **原型快速**：从聊天机器人开始，快速迭代
3. **试错成本低**：先演示效果，再逐步完善
4. **LLM能力探索**：不断尝试模型能做什么

#### AI驱动全栈开发框架思路
1. **领域驱动**：先定义领域模型和API契约
2. **架构先行**：先设计整体架构，再逐步实现
3. **质量保障**：测试覆盖、代码审查、持续集成
4. **渐进增强**：在现有系统基础上逐步加入AI能力

---

### 维度4：安全与权限

#### Harness Engineering思路
- **权限模型**：Claude Code的七层安全机制
- **执行环境**：沙箱隔离
- **审计追踪**：工具调用记录
- **用户控制**：频繁的权限弹窗（用户批准约93%）

#### AI驱动全栈开发框架思路
- **权限模型**：完整的RBAC权限体系
- **执行环境**：WASM沙箱 + 多层权限控制
- **审计追踪**：完整的操作日志和安全审计
- **数据治理**：敏感数据脱敏、数据可追溯链路

---

### 维度5：工具与技能管理

#### Harness Engineering思路
- **工具定义**：Schema形式定义，注入到LLM上下文
- **工具执行**：沙箱环境执行，结果格式化
- **技能**：Claude Code的Skills机制（注入领域指令）
- **扩展**：MCP服务器、Hooks、插件

#### AI驱动全栈开发框架思路
- **技能标准**：AgentSkills开放标准，SKILL.md文件格式
- **技能执行**：WASM组件模型，渐进式加载
- **工具集**：完整的内置工具（文件系统、网络、CLI等）
- **多语言**：支持多种编程语言编写技能

---

### 维度6：上下文与记忆管理

#### Harness Engineering思路
- **上下文策略**：
  - 压缩：对话历史摘要
  - 观察屏蔽：隐藏旧工具输出
  - 即时检索：动态加载数据
  - 子agent委托：压缩摘要返回
- **记忆层次**：
  - 轻量级索引（始终加载）
  - 按需拉取的详细主题文件
  - 只通过搜索访问的原始记录
- **Claude Code示例**：CLAUDE.md项目文件、自动生成的MEMORY.md文件

#### AI驱动全栈开发框架思路
- **RAG搜索**：混合密集+稀疏搜索（向量嵌入 + BM25与RRF融合）
- **交叉编码器重排序**：提高精度
- **查询理解**：意图分类和实体提取
- **上下文压缩**：token高效输出
- **状态持久化**：数据库存储，完整的状态管理

---

### 维度7：错误处理与可靠性

#### Harness Engineering思路
- **错误分类**：
  - 瞬时错误：带退避重试
  - LLM可恢复错误：将错误作为ToolMessage返回
  - 用户可修复错误：中断请求人工输入
  - 意外错误：向上冒泡用于调试
- **Stripe经验**：重试次数上限设为两次
- **成功率计算**：每步成功率99%的10步流程，端到端成功率约90.4%

#### AI驱动全栈开发框架思路
- **传统错误处理**：异常捕获、日志记录、监控告警
- **幂等性保证**：API调用幂等设计
- **降级策略**：关键路径保留人工Fallback
- **熔断机制**：超额预警和熔断
- **质量闸**：CI中设置静态检查、测试覆盖、性能阈值

---

### 维度8：性能与成本

#### Harness Engineering思路
- **Token优化**：上下文压缩、懒加载工具
- **Vercel经验**：从v0中删除80%的工具，结果更好
- **Claude Code**：懒加载实现95%的上下文减少
- **成本敏感**：每步都要调用LLM，成本累积

#### AI驱动全栈开发框架思路
- **高性能服务器**：支持1000+并发连接，平均响应时间<50ms
- **缓存策略**：内置缓存管理器，数据库连接池
- **成本控制**：平衡LLM调用与传统处理
- **资源监控**：资源使用监控和配额管理

---

### 维度9：可观测性与调试

#### Harness Engineering思路
- **LangGraph**：检查点、时间旅行调试
- **Claude Code**：git提交作为检查点，进度文件作为草稿纸

#### AI驱动全栈开发框架思路
- **日志系统**：结构化日志
- **指标监控**：性能指标、业务指标
- **分布式追踪**：链路追踪
- **可解释性**：AI生成代码包含决策路径

### 维度10：生态系统与集成

## 优势劣势对比表

| 维度 | Harness Engineering思路 | | AI驱动全栈开发框架思路 | |
|------|-------------------------|------|----------------------|------|
| | 优势 | 劣势 | 优势 | 劣势 |
| **快速原型** | 快速从聊天机器人开始，迭代速度快 | 从演示到生产的鸿沟巨大 | 架构先行，质量有保障 | 前期投入大，速度相对慢 |
| **模型能力** | 最大化释放LLM能力，探索边界 | 过度依赖模型能力 | 平衡模型与传统技术 | 可能限制模型潜力 |
| **可维护性** | 迭代灵活，易于尝试新想法 | 代码结构可能混乱，测试困难 | 清晰的分层架构，测试覆盖全 | 架构变更成本高 |
| **安全性** | 七层安全机制，沙箱隔离 | 用户审批疲劳，攻击面较大 | 完整RBAC，数据治理，审计追踪 | 配置复杂，易用性可能受影响 |
| **性能** | 上下文压缩优化，懒加载 | 每步都调用LLM，成本高，延迟大 | 高性能服务器，缓存策略 | 传统架构的性能瓶颈 |
| **可观测性** | 检查点、时间旅行调试 | 静默失败问题严重 | 完整日志、指标、追踪 | 系统复杂，监控成本高 |
| **生态系统** | AI生态活跃，社区强大 | 与传统系统集成困难 | 多语言SDK，传统生态集成好 | AI生态相对较小 |
| **成本控制** | Token优化策略好 | 累积成本高 | 平衡LLM调用，成本可控 | 前期基础设施投入大 |
| **人才需求** | 需要提示词工程师、LLM专家 | 人才稀缺，成本高 | 需要全栈工程师、架构师 | 人才相对充足 |
| **国产化** | 依赖国外大模型厂商 | 数据安全、自主可控性差 | 仓颉语言、国产技术栈 | 需要建立自己的生态 |
| **长期演进** | 模型改进时Harness可简化 | 与特定模型耦合深 | 架构稳定，可逐步增强 | 架构升级周期长 |

## 适用场景分析

### Harness Engineering思路最适合的场景

1. **创新探索型项目**
   - 需要快速验证AI能做什么
   - 项目目标不明确，需要探索
   - 例如：AI编程助手、创意工具

2. **以AI为核心的产品**
   - LLM能力是产品核心竞争力
   - 例如：Claude Code、GitHub Copilot

3. **原型验证阶段**
   - 需要快速演示效果
   - 获得用户反馈后再决定是否投入生产

4. **单一任务场景**
   - 任务边界清晰，不需要复杂的业务逻辑
   - 例如：数据查询、文档分析

5. **已有系统的AI增强**
   - 作为SDK或服务集成到现有系统
   - 不需要重构整体架构

---

### AI驱动全栈开发框架思路最适合的场景

1. **企业级应用**
   - 需要完整的业务功能、权限管理、审计追踪
   - 例如：ERP、CRM、OA系统

2. **全栈应用开发**
   - 需要前端、后端、移动端一体化
   - 例如：电商平台、SaaS应用

3. **数据安全要求高**
   - 需要自主可控、数据不出境
   - 例如：政府、金融、医疗行业

4. **长期演进项目**
   - 需要稳定的架构支撑长期发展
   - 需要逐步增强AI能力

5. **传统系统现代化**
   - 在现有系统基础上渐进式加入AI
   - 保留现有投资，降低迁移风险

---

### 混合策略

实际上，很多项目可能需要结合两种思路的优点：

1. **前期用Harness思路快速验证**
   - 快速原型，探索AI能力边界
   - 获得用户反馈

2. **后期用全栈框架思路生产化**
   - 重构架构，保证质量、安全、性能
   - 建立完整的运维体系

3. **分层设计**
   - 核心业务用全栈框架
   - AI交互层用Harness思路

## 总结

### 核心差异总结

| 方面 | Harness Engineering | AI驱动全栈框架 |
|------|---------------------|----------------|
| **核心定位** | LLM是核心，Harness是操作系统 | 系统是核心，AI是驱动子系统 |
| **设计哲学** | 最小脚手架，最大操作Harness | 领域驱动，架构先行 |
| **开发模式** | 提示词驱动，快速迭代 | 契约驱动，质量保障 |
| **智能来源** | 主要来自LLM | LLM + 领域知识 + 业务逻辑 |
| **安全策略** | 七层防护，用户审批 | RBAC + 数据治理 + 审计 |
| **性能优化** | Token优化，上下文压缩 | 缓存 + 连接池 + 传统优化 |

### 未来发展趋势

1. **Harness Engineering的演进方向**
   - 模型能力提升，Harness变薄
   - 更多功能内联到模型中
   - 标准化Harness组件
   - 更好的可观测性和调试工具

2. **AI驱动全栈框架的演进方向**
   - 更好地集成Harness工程最佳实践
   - 更智能的代码生成和辅助开发
   - 更完善的AI能力治理体系
   - 更丰富的AI原生组件库

3. **两者的融合**
   - Harness组件可能成为全栈框架的一部分
   - 全栈框架可能提供更友好的Harness开发体验
   - 标准化的接口和协议促进互操作

### 给项目方的建议

1. **选择思路时考虑**
   - 项目性质（创新探索 vs 企业应用）
   - 团队能力（AI专家 vs 全栈工程师）
   - 时间要求（快速验证 vs 长期演进）
   - 安全要求（数据敏感程度）
   - 成本预算（LLM调用成本 vs 基础设施投入）

2. **不要非此即彼**
   - 可以分阶段采用不同策略
   - 可以在不同模块采用不同思路
   - 关键是理解两种思路的本质差异和适用场景

3. **关注长期价值**
   - Harness思路适合快速验证，但生产化需要更多投入
   - 全栈框架前期投入大，但长期维护成本更低
   - 建议根据项目生命周期动态调整策略
